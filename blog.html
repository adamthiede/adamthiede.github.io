Rolling or Static

[link—standalone]

Software updates happen. Your operating system and its programs get updates all the time, and you should install those updates. Sometimes they provide bug fixes, security patches, or new features. Sometimes these changes are "breaking changes", or paradigm shifts that break workflows or existing configurations.

Rolling release is a model that supposes you take all the updates as they come, immediately, with little modification. This is like installing a program (say, a specific release of Docker, LibreOffice, or GIMP) and keeping it updated regularly as the creator pushes out new releases. Simple enough to understand - you get the changes as they come and update frequently but on no particular schedule. Rolling releases fit with the saying "every update is a security update". When software interacts in interesting ways, sometimes a simple bug can turn into a massive security hole, so it is important to keep systems and software patched.

A good example of a rolling release operating system is Arch Linux - packages arrive in the repositories as soon as they're released, and Arch doesn't have any particular release schedule. You just install once, and update forever. But you must update frequently or you'll be without security patches. Sometimes the software doesn't work too well together - it's not tested or specifically modified to integrate well with the rest of the system.

Static release is a model built for the enterprise, but beneficial to anyone who doesn't like surprises. You pick a base version of a program (LibreOffice 6.4 for example), and if there are any bug fixes or security patches that affect that particular version, you apply them. New versions come out, but being cautious, you save your major version upgrades for all your software for a specific interval. Example: every year in April, you go through the list of software you use, see if there are any new releases, and if those new versions have introduced breaking changes to your workflow. You then integrate the new software if you desire, apply fixes to the old software if you must keep that version, and carry on.

Static release is a harder model to understand. It requires a lot of work. Thankfully many people do that work for you. If you use a Linux-based operating system, you can get a number of static release systems. Red Hat Enterprise Linux releases a new version of their operating system every 4 years or so. That release is supported for ten years. You can continue using the same stuff you were using in 2014 (RHEL7) with the same software it came with, only with regular security fixes "backported" to the versions that were chosen at the time of release. Specific versions are chosen and tested against each other to make sure they work well. They bundle an old version of the Linux kernel, old versions of desktop programs, and old versions of server software. But this means users and businesses don't get surprised by random changes, and can rely on the system being stable and secure for a long time. They can plan their own software or infrastrucutre around it, since it doesn't change out from under them. Some long-term support releases of operating systems, like Ubuntu, are more frequent - they release every 2 years on the dot, and enterprises (and stability-concerned users) can count on this schedule. Every 2 years in May or June they can upgrade to the new version, and plan to do all their troubleshooting then, as opposed to random times throughout the year, when they're trying to get work done.

Some operating systems use a Semi-rolling model. Fedora, the "upstream" of RHEL (what RHEL is based on, before making significant changes) releases a new version every 6 months. These releases keep some desktop and server software at a single version for those six months, only applying bug fixes. But other software is updated as the updates come - the Linux kernel follows upstream releases closely, while GNOME, PHP, Python, gcc, and other programming tools and server software promise to only change every 6 months. This gets benefits of both sides - most of your stuff is as up to date as possible, but critical software doesn't change out from under you. You know every 6 months you'll have to examine the major upgrades for breaking changes, but the changes between bi-yearly release are usually minor. The reason for these cycles is so that the project can pick a date to do major changes and have milestones.

Rolling release models are not for the faint of heart. They require constant vigilance to make sure upcoming changes don't break things. Every update has the potential to render your desktop or web server unusable. You don't get to pick when things break or when there are major changes. You must update freuquently and that requires a lot of extra CPU usage and data usage, so not great for metered networks.

Static releases can be just as bad - old software won't work with new tools you want to use. When new hardware comes out, the old software won't support it, and you may have to wait months or years before the changes you're waiting for land in the operating system you're using. You get to choose when you do full release upgrades, but often times, it takes days or weeks to iron out all the things that have broken or changed releases. 2 to 5 years is a huge jump in version numbers and having been out of the loop will mean you have a lot of catching up to do.

Both releases are valid models. Home users or enthusiasts are likely more interested in rolling models - newer software as it comes usually causes no issues, fixes bugs, and provides more up to date hardware support. Entereprises, developers, and cautious users stick to static releases - less surprise, nothing changing out from under you in the middle of a project, and a solid long-term support strategy. Semi-rolling, in my opinion, gives you the best of both worlds without the downsides. But in the end it's up to preferences, goals, and needs of the individual.

Modern Cars

[link—standalone]

I went car shopping recently. In the latest episode of "they don't make 'em like they used to", I level complaints at modern automobiles.

  • Many cars have built-in touchscreens that replace functionality that used to be buttons and knobs. It is easier and safer to memorize where your buttons and knobs are, and not have to look at them to change volume, temperature, and other system features about your car. Cars like Teslas, where the entire control system is a big touchscreen, are more dangerous than my decade-old Toyota Camry - I know where all the knobs are, can find them by feel, and can adjust them without looking. (How is using a car touchscreen different than looking at your smartphone while driving, an activity illegal in many states?)
  • Newer cars often include computers with wireless connectivity. Obviously integrating a computer with opaque software that the user doesn't control is bad, but what's even worse is the security implications - often these systems ship with serious security vulnerabilities and never get fixed, allowing bad actors remote control of the vehicle or tracking of its location. The more complex the system, the more chances for exploitation.
  • Most vehicle's models are quite a bit larger than the same model that was shipping 10 or 15 years ago, with no increase in functionality.
  • Some new safety featues, like backup cameras or adaptive cruise control, are functional but not yet reliable, and are expensive to repair when damaged. It's a "false sense of security" thing. People are less likely to look around their car when they have a backup camera. You've also likely seen news of Tesla owners crashing their vehicles because "full self driving" features aren't exactly full self driving yet.
  • Personal complaint - CD drives are all but gone. You now have to look at a touchscreen to select music from an Internet service, or via smartphone integration. It's a global waste of energy and brings me back to the first point - it's easier to manipulate a CD case by feel than a touchscreen.

Cars are more safe than ever. They're more energy efficient and have a bunch of great new features. But not all of the features are good, and we should be discerning about what we as consumers accept. More safety and security - fantastic! More Internet connectivity, "smart" features, and controls transformed from buttons into touchscreens - get it out of here!

This edition of "things aren't like they used to be" brought to you by the 30-Year-Old Boomer Association of America.

Good Tools and Bad People

[link—standalone]

I have a love-hate relationship with technology/culture news site The Verge. On the one hand, the people who founded it are cool and interesting. They tend to cover stuff pretty factually in the realm of technology - their journalists are sharp and funny. On the other hand, they seem to ignore some basic facts in service of eradicating free thought, especially if it's thought they don't like. They've published a piece about Signal's lack of moderation of their service. It is well written and offers real concerns about a real problem - bad people can use Signal. They can send whatever they want to each other and Signal isn't doing a damn thing about it!

But it's actually not a problem. Signal makes a tool. People use the tool. Signal can't monitor who is using it, how it's being used, and what's being shared there. The main reason people use Signal at all is because Signal can't see anything on their service, and you can trust that this is the case, since it's fully Open Source. How would they "moderate"? How would they censor content? How do they even know at all what's being published on their messaging service without destroying the one and only reason people use it and trust them?

Almost two years ago, I was concerned about Facebook's idealism and how that would eventually get them into trouble. Fast forward two years - last month they joined other social media sites in banning the President of the United States of America from their suite of websites. Because of the way Facebook is constructed, they can see every word you type on their system. Because they have the ability to moderate, they have the responsibility to do so, and they do it according to their own company's values. People who don't want their chat app to have a value system (especially one that differs from theirs) pick a chat app without a value system, like Signal. It does what it says it does - gets your messages from one end to the other with no spying in between. It is reliable, trustworthy, and simple. Signal is a high-quality tool.

If you make a good tool, and that tool also doesn't spy on you or hurt you, then people are going to want to use it. A portion of the people in the world are bad people. I use my hammers and shovels as tools to get work done. There's probably someone in the world who uses hammers to kill puppies and shovels to kill kittens. That's a despicable way to use these tools, and not what they are designed to do. But if you make a high-quality versatile tool, people are going to misuse it. But that doesn't mean you should abandon making better hammers or sturdier shovels.

There are probably more good people using Signal to chat with their family and have normal discussions than there are bad people plotting to overthrow their perfectly normal government. If bad people use the service, that's just a price you have to be willing to pay for making a high quality product that's also respectful of the user's basic human rights. Sometimes making the world a better place for good people also makes it a better place for bad people. Overall, it is worth it to help the good people. Sometimes bad people use good tools. That's just how it works.

Moralizing Software? Start at Home

[link—standalone]

Quick thought: if you want to stop using Gmail for vague moral reasons, think about what's going to have a greater cost/benefit ratio. If you want to change your software or network services setup for moral reasons ("Google/Facebook/$BIG_COMPANY is spying on me") then start with the things you use most often, and start with the hardware in your home. Get rid of the Amazon Echo and other "smart devices". Use Linux. Put your smartphone in the other room and turn it off when you're not using it. Use a password manager like KeePassXC and keep things offline on local storage instead of on the Internet. Encrypt your devices' storage. Don't send personal or sensitive communications over email, use Signal or Matrix or some other encrypted method.

Swapping out Google for Fastmail or whoever is going to have such a small impact on your overall privacy, chances are you won't follow through with it, you won't move all your accounts over, and it'll be a headache. Do something that's going to have immediate, real effects. Ditching Gmail has far too low of a cost-benefit.

Protocols, Not Platforms

[link—standalone]

Over the past year I've been asked by family members and friends about platforms, apps, and services. "Should I use this app?" "Is this social media site secure?" "What do you think of this email provider?"

These are the wrong questions to ask.

The reason these questions are being asked is that people are getting more concerned with security, privacy, and longevity of the systems that they use. If a system is not secure and private, they're afraid they can't express their opinions or talk candidly to friends and family via the service. They've also noticed that services they use go away sometimes, or are offline for long periods of time. People don't want to have to constantly switch from SMS to iMessage to WhatsApp to Viber to Telegram to Signal, because change is hard, and shifting your whole social group to use a different system is a very difficult long-term effort.

The right question to ask is, "What tools can we use that leverage open, extensible, secure, protocols, ensuring longevity and privacy?

To understand what I'm talking about, you need to think about email. This is the extensible open platform people have the most experience with. You can send an email from your @gmail.com account, via the iPhone's mail app, to your friend's @yahoo.com address, who checks his email in Microsoft Outlook on a Windows PC. Which app you use doesn't matter here - you could use any number of email programs ("clients") to interface with your account. Gmail and Yahoo mail are just two fish in the sea of email servers. Email the protocol has been around for decades, and will work for decades more, even if Gmail and Yahoo go away. You can even run an email server in your house on an old spare computer, or on a rented cloud virtual machine, for less than $5 a month. This is the power of an extensible, open, decentralized protocol.

Focusing on protocols (the method and format of the data transmission) instead of platforms (who is holding and transmitting your data) puts the power in your hands. It also puts the responsibility in your hands. You can choose an email provider, or go about the herculean process to host email yourself. Hosting your data and services yourself requires a fair amount of knowledge, but it also makes you self-reliant, resilient to censorship, and insulated from the constant churn of platforms that exist, and then suddenly don't a few years later. Your server's downtime (for updates, etc.) happens on your terms, not Google's or Amazon's.

Extensible, open software I recommend:

  • Matrix Protocol for secure communication. Encryption built-in, many different clients available, servers are easy to host, and most importantly, it's open-source and respects your freedom.
  • XMPP/Jabber is the universal messaging standard. You've probably used it before without knowing it. Layering OMEMO or OTR encryption on top of XMPP is like a better version of Signal. Check here for a list of servers that support all the appropriate extensions required to run a secure XMPP conversation, or just start your own.
  • ActivityPub Protocol and its implementations Mastodon and Pleroma. Decentralized DIY social media. No algorithms, no censorship or oversight. Each server can choose its own rules.
  • PeerTube is DIY YouTube. It also implements ActivityPub, so people can comment on videos with their Pleroma or Mastodon accounts.
  • NextCloud is DIY Dropbox, with integrated contacts/calendars. Calendar and Contact syncing is done over the open DAV protocols, CalDAV and CardDAV, and comes for free with many email setups.
  • MailInABox if you must self-host your own email. Easy to set up, and keep secure. Also sets up a small Nextcloud instance.
  • This list contains more resources I didn't cover. Most of these things will integrate well with each other.

Signal had an outage the other day that lasted for over 24 hours. Messages were unreliable or didn't go through, and forget about calls. This happened because Signal is a single entity, and the only way you can use Signal is through their single point of failure - them. Decentralized protocols federate the responsibility to users, so if your server is down, your parents can still talk to their friends on different servers.

A couple guidelines to live by:

  • Open Source is a must.
  • If it requires a smartphone app, it's not worth your time.
  • Client and Server should have multiple implementations that run on many operating systems - you should be able to host the service, and interact with it, from any computer.

In a world where you can't trust surveillance-addicted megacorps to respect your privacy, and you can't trust that scrappy moralizing startups won't either fail or get acquired by the megacorps, the only reasonable choice is to trust individuals. The best individuals to trust are yourself, your family, and your friends - if every family hosted their own services, outages would be a thing of the past, privacy would be table stakes, surveillance would be much more difficult, and we'd all have control of our data. So do some research, and continue to ask the right questions.
