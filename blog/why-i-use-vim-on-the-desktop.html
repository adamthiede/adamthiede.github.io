Why I Use Vim On The Desktop

[link—standalone]

Vim is popular because it’s always where you need it. I use it because I’ve never found a place where Vim wasn’t there first or incredibly easy to install.

I would argue that Vim is best used as close to default as possible, with as few plugins as possible. Most of the quick one-off edits I need to do happen on remote servers. Since they’re cattle, not pets, we keep the images as default as possible. On most modern Linux systems, Vi(m) is present, and the default. On the desktop, I do a lot of interacting with git repos, and the best way to do that is in the terminal. Editing files in the same window I’m managing files and repos in is quicker and more ergonomic for me.

When I git commit, Vim is there. When I visudo, Vim is there. Any remote system I need to make a quick change on, Vim is there. When I pull down an Alpine Linux container or spin up an OpenBSD server I can be sure that Vi will be there, and Vim will be installable from repos, if not present already.

So why not use it for editing on my desktop as well? It’s certainly as extensible and powerful as Atom/TextMate/VSCode. There’s no end to the functionality, too; I pick up new features a couple times a year and add those to my tool belt. I remember a lot of keyboard shortcuts and movements for Vim, and the “Vim modes” in these other editors always ends up being half baked to the point of uselessness, so I go back to Vim.

Running default is important to me. My vimrc is less than 50 lines, and I use three plugins. They assist with writing prose and don’t hinder me by not being there on servers.

If you’re only editing on the desktop, use TextMate or Gedit or VSCode or whatever. They work great. But in my workflow I value consistency, and I frequently edit on systems I can’t customize, so why not get comfortable with the tool that I know is going to be everywhere?
